# פרומפט: בניית חבילת הוראה מלאה לקורס Vibe Coding (Teaching Package)

הוראה: העתק/י את כל הטקסט שבתיבה למטה (מ-"אתה עוזר..." ועד הסוף) והדבק/י אותו כהודעה ל-Claude. מומלץ להריץ שיעור אחד בכל פעם. אם מריצים בסביבה עם גישה לקבצים (כמו Claude בסביבת Cowork/Claude Code) - קלוד יכול ליצור בפועל את מבנה התיקיות והקבצים המתואר למטה, לא רק להדפיס טקסט.

## סטנדרט צפיפות תוכן ואינטראקטיביות (חובה בכל שיעור)

לקח מבניית שיעורים 1-2: תוכן שנכתב רק כרשימת בולטים תמציתית "נגמר" בפועל הרבה לפני הזמן שהוקצה לו (סטודנט קורא בולט ב-15 שניות; מרצה שמסביר בולט בלי תוספת אמיתית מסיים תוך כמה דקות, לא תוך רבע שעה). כדי שכל שיעור באמת ימלא את 90 הדקות שהוקצו לו:

1. **כל בלוק תוכן צריך פעילות אינטראקטיבית אחת לפחות** (שאלת כיתה + זמן לתשובות, פעילות בזוגות של 2-3 דקות, או "הרימו יד"). פעילויות תמיד לוקחות יותר זמן אמיתי מהסבר פרונטלי - זה לא "בזבוז זמן", זו הדרך האמינה למלא זמן הרצאה נכון.
2. **כל שיעור צריך לפחות "נקודת גישור" אחת למסלול בינה מלאכותית** - קטע ב-lesson.md (ומשתקף גם ב-instructor_notes.md) שמקשר את הנושא לרקע שהסטודנטים כבר מביאים מהמסלול שלהם (Colab/Kaggle notebooks, שימוש ב-ChatGPT/Claude לקוד עזר מחקרי, מושגי ML/AI רלוונטיים). זה גם ממלא זמן אמיתי (הסטודנטים מזהים את עצמם בדוגמה, יש להם מה להגיד) וגם הופך את הקורס לרלוונטי יותר לקהל הספציפי הזה.
3. **בכל נושא שיש לו יותר מדוגמה אחת בעולם האמיתי - תנו שתיים.** לדוגמה: בשיעור 1 מקרה הבוחן Gemini CLI→Antigravity קיבל דוגמה שנייה (שינוי מסלול GitHub Copilot Student) בשקף נפרד, כי דוגמה אחת בלבד לא מילאה את הזמן שהוקצה לה. חפשו באינטרנט דוגמאות עדכניות נוספות במקום להסתפק בדוגמה יחידה כשיש זמן שצריך למלא.
4. **בסוף כל instructor_notes.md, כללו "חלוקת זמן מעודכנת"** - פירוט מדויק של כמה דקות לכל בלוק תוכן, שמסתכם לסביבות 90 (מותר 90-95, לא פחות מ-85). אם הסכום נמוך משמעותית מ-90 - זה סימן שצריך להוסיף עוד פעילות/דוגמה/נקודת גישור, לא רק "לדבר לאט יותר".
5. **אל תוסיפו תוכן על ידי הרחבת כל בולט לפסקה ארוכה** - זה עובד הפוך: שקפים עמוסים בטקסט גרועים פדגוגית. התוכן הנוסף שייך לפעילויות, לדוגמאות נוספות, ולקטעי lesson.md - לא לעומס טקסט על השקף עצמו.
6. **כל שיעור צריך 16-20 שקפים לפחות** (לא 10-12) כדי למלא 90 דקות הרצאה בעומק אמיתי. הדרך הנכונה להגיע לזה היא לא לנפח בולטים קיימים, אלא להוסיף שקפי תוכן חדשים מהסוגים הבאים:
   - **שקפי סטטיסטיקה** (Stat Callouts - מספר גדול + תווית קטנה) עם נתונים אמיתיים ומצוטטים - חפשו באינטרנט נתוני שוק, סקרים, ובנצ'מארקים עדכניים (DORA, Stack Overflow Developer Survey, METR, SWE-bench/Terminal-Bench, נתוני אימוץ של כלים) לפני שכותבים שקף כזה. כל שקף נתונים חייב ציון מקור בתחתית השקף.
   - **שקפי מקרה בוחן נוספים** - כשיש יותר מדוגמה אחת בעולם האמיתי לנושא, תנו שתיים (ראו שיעור 1: מקרה Gemini CLI + מקרה Copilot Student; שיעור 1: Stripe/TELUS/Zapier).
   - **שקפי "הנתונים תומכים בזה"** שמקשרים בין קונספט שכבר לומד (Workflow, Best Practices) לבין נתון אמיתי שמוכיח אותו במספרים - אלה השקפים הכי אפקטיביים פדגוגית כי הם הופכים טענה לעובדה מגובה.
   - **שקף מקורות** בסוף כל שיעור (לפני/אחרי הסיכום) שמרכז את כל המקורות שצוטטו.
   - חשוב: אין לבדות נתונים. כל מספר על שקף חייב להגיע מחיפוש אמיתי באינטרנט ולהיות מתועד ב-references.md. אם לא מוצאים מקור אמין לנתון - לא כותבים אותו כעובדה, לכל היותר כהערכה מנוסחת בזהירות.
   - כשמציגים בנצ'מארקים טכניים (כמו SWE-bench) - תמיד לכלול אזהרת מתודולוגיה קצרה (שיטות מדידה שונות, שונות בין הרצות) כדי לא להציג מספר בודד כאמת מוחלטת. זה גם תוכן וגם מודלים חשיבה ביקורתית.

## דרישה קריטית: בדוק מה עדכני היום
לפני שאתה כותב תוכן לשבוע שנוגע לכלים, מחירים, מסלולים חינמיים, או

2. איזו אופציה חינמית (או הכי זולה) קיימת להתחלה עבור סטודנטים,
   כולל הגבלות (מכסות, אימות סטודנט וכו').
3. אם משהו עלול להשתנות עד מועד ההוראה בפועל, ציין זאת במפורש
   בהערת מרצה קצרה בתחתית השקף הרלוונטי.

## דרישות מדויקות לכל קובץ בתיקיית השיעור

**lesson.md** - חומר קריאה עצמאי לסטודנט, טקסט רציף (לא בולטים) שמסביר
את נושאי השיעור בסדר הגיוני, ברמה שמאפשרת לסטודנט שהחסיר שיעור להבין
לבד. אורך: כ-1500-2500 מילים.

**slides.md** - תוכן המצגת בפועל, מוכן להמרה ל-Marp/PowerPoint. **16-20
שקפים לפחות** (ראו "סטנדרט צפיפות תוכן" למעלה - זה לא אופציונלי): שקף
פתיחה (מספר שיעור, כותרת, 3-4 שאלות מנחות) → שקפי תוכן (כותרת קצרה +
בולטים, לא פסקאות - אחד לכל נושא מרשימת הנושאים של השבוע) → שקפי
נתונים/סטטיסטיקה מבוססי מקורות אמיתיים משולבים בין שקפי התוכן → שקף
הדגמה חיה → שקף מעבר לתרגול → שקף סיכום (3 Takeaways + חיבור לשבוע
הבא) → שקף מקורות. סמן כל שקף עם `---` כמפריד (תואם Marp).

**instructor_notes.md** - מדריך הפעלה למרצה, מקביל לכל שקף ב-slides.md:
מה להגיד/להרחיב, תזמון משוער לכל בלוק, שאלות לדיון שכדאי לזרוק לכיתה,
ומוקשים נפוצים (איפה סטודנטים בד"כ נתקעים או מבינים לא נכון).

**exercises.md** - הנחיות לתרגול המודרך (חלק ה-45 דקות): מטרת התרגול,
Prompt/ים לדוגמה שהסטודנט יכול להשתמש בהם כנקודת פתיחה מול כלי ה-AI,
ותוצר מצופה בסוף 45 הדקות (Definition of Done קצר וברור, לא "פתרון").

**solutions.md** - לא פתרון יחיד "נכון". מכיל: (א) rubric/קריטריוני
קבלה - רשימת מה שמבדיל תרגול תקין מתרגול חלקי, (ב) implementation
אחת לדוגמה (קוד עובד), כהפניה למרצה בלבד - לא לחלוקה מראש לסטודנטים.

**demo/** - קוד בר-הרצה אמיתי (קבצים מלאים, לא קטעים) שמדגים את תרחיש
ההדגמה החיה שמוזכר ב-slides.md. המרצה צריך להיות מסוגל להריץ את זה
ישירות מול הכיתה. אם הדגמה כוללת אינטראקציה עם כלי AI - כלול גם את
הפרומפט המדויק שיוקלד בזמן אמת כקובץ `demo/prompt.md`.

**diagrams.md** - כל דיאגרמה נדרשת (ארכיטקטורה, ERD, flow) ככתיב
Mermaid בתוך בלוק קוד. אל תנסה לייצר קובצי תמונה - LLM טקסטואלי לא
מסוגל לכך; קוד Mermaid ניתן לרינדור בקלות בכל כלי מצגות/Markdown מודרני.

**references.md** - קישורים ומקורות רלוונטיים לנושאי השבוע, כולל הערה
אם מדובר במידע (כלים/מחירים) שצפוי להשתנות ולכן כדאי לבדוק מחדש קרוב
למועד ההוראה בפועל.

## סגנון וטון
- עברית לקהל דובר עברית; מונחים טכניים (Context, Debugging, Deploy
  וכו') נשארים באנגלית כמקובל בתעשייה.
- שקפים תכליתיים: בולטים קצרים, לא משפטים ארוכים. הערות הדובר יכולות

- דוגמאות קוד קצרות ורלוונטיות ל-Stack של הקורס (Python/FastAPI +
  React), לא דוגמאות מנותקות מהפרויקט.
- קהל היעד הוא תעשייה וניהול - כשמסבירים קונספט טכני, כדאי אנלוגיה
  או חיבור לחשיבה עסקית/תהליכית שהם מכירים, בלי להנמיך את רמת התוכן.

## רשימת השבועות (נושאי-ליבה בלבד לפי הסילבוס המצומצם ל-90 דק')

חשוב: לכל שבוע יש כאן רק 3-4 נושאי-ליבה שמיועדים ל-90 הדקות. אם צוין
"קריאה עצמאית" - זה תוכן שעדיין חשוב, אבל שייך ל-lesson.md ולא לשקפי
ההרצאה/הדגמה החיה. אל תדחוס את נושאי הקריאה העצמאית לתוך slides.md -
הם שייכים אך ורק ל-lesson.md, בדיוק כדי שההרצאה תישאר ריאלית ל-90 דקות.

שבוע 1 - Introduction to Vibe Coding (ליבה): מהו Vibe Coding + Evolution
of Software Development; יתרונות/חסרונות/מגבלות מודלי שפה; Workflow של
מפתח מודרני; מקרה בוחן Gemini CLI→Antigravity ("Workflow לא כלי");
הדגמת בניית אפליקציה ב-20 דקות. תרגול: Todo App באמצעות AI.

שבוע 2 - AI Development Environment (ליבה): VS Code + Git/GitHub -
הקמת סביבה; Claude Code כדוגמה לכלי Agentic מוביל - בדוק סטטוס עדכני;
כלים חינמיים להתחלה (בדוק עדכני); Best Practices. קריאה עצמאית: השוואת
מודלי שפה מפורטת, יסודות Docker (יילמד בעומק בשבוע 11). תרגול: הקמת
סביבת עבודה ובחירת כלי AI חינמי.

שבוע 3 - Context Engineering (ליבה): Context Engineering; CLAUDE.md/
AGENTS.md בהדגמה מעשית; Prompt vs Context; Specification-Driven
Development. קריאה עצמאית: Project Rules, Memory, Documentation
conventions. תרגול: בניית Context לפרויקט.

שבוע 4 - Working with Existing Code (ליבה): Reading Code + Reverse
Engineering; Refactoring; Documentation Generation; Safe Code Changes.
תרגול: הוספת Feature למערכת קיימת.

שבוע 5 - AI Agents (ליבה): Agentic Coding; Planning & Task
Decomposition; MCP (Model Context Protocol - בדוק עדכני) & Tool
Calling; Multi-Agent Workflow (הצצה קצרה בלבד). קריאה עצמאית: MCP spec
מעמיק, תבניות Multi-Agent מתקדמות. תרגול: בניית Agent שמבצע משימה
מורכבת.

שבוע 6 - Designing Before Coding (ליבה): Requirement Analysis + User
Stories; System Design + Architecture Diagrams; Database Design -
יסודות (טבלאות, קשרים, שאילתה בסיסית, כי לקהל אין רקע DB); REST API
Design. קריאה עצמאית: Design Patterns (יידונו אגב אורחא בזמן Code
Review בשבועות 9-10, לא כהרצאה נפרדת). תרגול: תכנון מערכת חדשה.

שבוע 7 - Building the Application חלק א' (ליבה): FastAPI - יסודות
Backend; CRUD מלא עם AI; Authentication - יסודות. תרגול: בניית Backend
+ CRUD בסיסי.
