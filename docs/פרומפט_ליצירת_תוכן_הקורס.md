# פרומפט: בניית חבילת הוראה מלאה לקורס Vibe Coding (Teaching Package)

הוראה: העתק/י את כל הטקסט שבתיבה למטה (מ-"אתה בונה חבילת הוראה מלאה..." ועד הסוף) והדבק/י אותו כהודעה ל-Claude. מומלץ להריץ שיעור אחד בכל פעם. אם מריצים בסביבה עם גישה לקבצים (כמו Claude בסביבת Cowork/Claude Code) - קלוד יכול ליצור בפועל את מבנה התיקיות והקבצים המתואר למטה, לא רק להדפיס טקסט.

**הערה:** שלושת הסטנדרטים המחייבים ליצירת תוכן (אפס-ידע-קודם, שקף עצמאי, צפיפות ואינטראקטיביות) נמצאים **בתוך** תיבת ההעתקה למטה, לא כהערות חיצוניות למסמך - כך שהם באמת חלים גם כשמעתיקים את הפרומפט לשיחת Claude חדשה ונפרדת.

## מבנה הפלט לכל שיעור

כל שיעור מיוצר כתיקייה עצמאית, עם **ארבעה קבצים בלבד** - אין lesson.md,
אין exercises.md, אין demo/, אין diagrams.md. הסיבה: כל אלה יצרו בפועל
תלות של הסטודנט/המרצה בקובץ נוסף מלבד המצגת (כדי לקרוא הסבר, כדי
להקליד פרומפט הדגמה, כדי לדעת מה התרגיל) - וזה בדיוק מה שהתגלה כטעות
חוזרת בבניית שיעור 1 (ראו "מצגת כמקור יחיד" למטה). **כל תוכן שהיה
מיועד לקבצים האלה חייב להיות מוטמע ישירות בתוך slides.md/ה-PPTX עצמו**
- כולל טקסט קריאה, פרומפט ההדגמה החיה במלואו, הנחיות התרגיל המלאות,
ותיאורי הדיאגרמות (כ-placeholder תמונה בתוך השקף הרלוונטי).

```
lesson_01/
├── slides.md            # תוכן המצגת - המקור היחיד לסטודנט. כולל הכל:
│                         # הסברים, פרומפט ההדגמה החיה, הנחיות התרגיל
│                         # המלאות, כל דיאגרמה כ-placeholder בתוך השקף
│                         # הרלוונטי. build_deck.js הוא הקוד שממיר
│                         # אותו בפועל ל-lesson_01_slides.pptx.
├── instructor_notes.md  # מדריך למרצה בלבד: מה להגיד, תזמונים, מוקשים
├── solutions.md          # רubric/קריטריוני קבלה לתרגיל + implementation
│                         # לדוגמה אחת מלאה (קוד מוטמע בקובץ עצמו,
│                         # לא בתיקיית demo/ נפרדת) - למרצה בלבד
└── references.md        # רשימת מקורות/קישורים מלאה - גיבוי לציטוטים
                          # שכבר מופיעים בתמצית על שקף "מקורות" במצגת
```

**מבחן ההכרעה אם משהו שייך למצגת או לקובץ נלווה:** אם סטודנט צריך את
זה כדי להבין/לבצע במהלך השיעור - זה בשקף. אם זה מיועד רק למרצה לפני/
אחרי השיעור (רוברית הערכה, תזמונים, קישורים גולמיים למקורות) - זה יכול
להישאר ב-instructor_notes.md/solutions.md/references.md, כי אלה לא
קבצים שהסטודנט אמור לקרוא בזמן אמת.

## שלושה סטנדרטים מחייבים לכל שיעור (לא אופציונליים, חלים על כל 13 השבועות)

**עקרון-העל שמעליהם, שחל על כל שקף בכל שיעור בקורס:** המצגת (קובץ
ה-PPTX/slides.md) צריכה להיות בנויה כך שחומר ההוראה שנמצא בה מספק
**ב-100%** - סטודנט שקורא **אך ורק** את המצגת (בלי lesson.md, בלי
מרצה, בלי שום מקור אחר) חייב להבין את כל החומר שהוצג, על בסיס מה
שכתוב שם בלבד, **ב-100%**, ולצאת **מומחה** בתוכן הזה. זו לא שאיפה
כללית - זו דרישת קבלה (acceptance criterion) לכל שקף שנכתב: כל מונח,
כל שם, כל נתון השוואתי, כל מספר נגזר, וכל שאלה שהשקף עצמו פותח - חייבים
תשובה/הסבר/הקשר מלא **על אותו שקף**, בלי לסמוך על ידע קודם, על שקף
אחר שהסטודנט אולי לא זוכר, או על lesson.md שהסטודנט כנראה לא יקרא. כל
שיטת הבנייה של השקפים (סטנדרטים 1-3 למטה) היא רק פירוט מעשי של
העיקרון הזה.

### 1. סטנדרט אפס-ידע-קודם

הקהל הוא סטודנטים לתעשייה וניהול, מסלול בינה מלאכותית — **מניחים שהם לא
מכירים אף מושג טכני-הנדסי, כולל הכלים הכי בסיסיים** (Git, GitHub, VS
Code, IDE, Terminal, וכן הלאה בהמשך הקורס: FastAPI, REST API, Docker,
וכו'). רקע ה-ML/AI המושגי שלהם (Colab, Kaggle, שימוש ב-ChatGPT/Claude)
שימושי כאנלוגיית גישור, אבל לא מחליף הסבר מלא של כל מושג הנדסת-תוכנה
מאפס.

**כלל מחייב: לכל מושג טכני חדש - שקף נפרד משלו (או קשת שקפים, ראו
סטנדרט 2), לא שילוב של כמה מושגים בשקף אחד.** לדוגמה: "VS Code, Git
ו-GitHub" הם שלושה מושגים נפרדים, כל אחד צריך שקף (או יותר) משלו:
- שקף שמסביר מה זה IDE בכלל (למה לא מספיק Notepad), ואז מה זה VS Code
  ספציפית.
- שקף שמסביר מה זו הבעיה ש-Version Control פותר (למה לא מספיק לשמור
  "v1, v2, v2_final, v2_final_REALLY")
- שקף שמסביר מה זה Git ספציפית - Repository, Commit - כמושגים נפרדים
  ומוגדרים.
- שקף שמסביר מה זה GitHub, ובמיוחד ההבדל בין Git (הכלי) ל-GitHub
  (השירות בענן) - זה בלבול נפוץ אצל מתחילים.

אותו עיקרון חל על כל מושג טכני בהמשך הקורס: REST API, Database,
Docker, Deployment, CI/CD - כל אחד מקבל שקף/קשת הסבר-מאפס משלו לפני
שמשתמשים בו כמובן מאליו.

**כל שקף שמסביר מושג חדש חייב לכלול placeholder מפורש לאיור** (בפורמט
`[תמונה: תיאור מדויק של האיור שיעזור להבין את המושג]`) - המרצה/היוצר
יכניס את האיור בעצמו, אבל התיאור צריך להיות ספציפי מספיק כדי לדעת
בדיוק איזה איור נדרש (לא "תמונה רלוונטית" אלא למשל "תמונה: איור שמראה
קובץ שנשמר ב-3 גרסאות שונות עם שמות מבולבלים, לעומת אותו קובץ עם
היסטוריית Git מסודרת").

### 2. סטנדרט שקף עצמאי (Self-Contained Slide)

**המבחן העליון של הסטנדרט הזה:** סטודנט שקורא אך ורק את קובץ המצגת
(בלי lesson.md, בלי מרצה, בלי הסברים בעל-פה) צריך לצאת **מומחה בכל
נושא שהמצגת נגעה בו** - לא רק "לא מבולבל" ולא רק "לא נשארו לו שאלות
פתוחות". זו השיטה המחייבת לבניית כל מצגת בקורס: בכל שלב בבנייה שואלים
"אם הסטודנט הזה יבחן מחר על התוכן הזה בהתבסס רק על מה שראה בשקף - הוא
יעבור?" אם התשובה לא ודאית - השקף חסר, לא רק "אפשר לשפר".

מבוסס על ניתוח מצגת-לדוגמה בתחום ה-Deep Learning (NN Optimization)
שנבחרה כמודל לחיקוי. הדפוס שם: כל שקף תוכן (לא שקפי מעבר/כותרת) עומד
בפני עצמו - סטודנט שקורא אותו בלי מרצה עדיין מבין את המושג ויודע מה
לעשות איתו, לא רק רואה שאלה או כותרת "באוויר". שקף כזה חייב לכלול, לפי
הסדר:

1. **שם/כותרת המושג.**
2. **הבעיה או השאלה שמניעה אותו** - עדיף לפתוח בבעיה קונקרטית ("איזה
   מודל טוב יותר?", "מה קורה כשכמה אנשים משנים קוד במקביל?") ולא
   בהגדרה יבשה - זה נותן למושג טעם ומוטיבציה לפני ההגדרה הפורמלית.
3. **הסבר אמיתי של המנגנון/רעיון** - משפטים מלאים שמסבירים למה/איך,
   לא רק שם המושג כבולט בודד. דיאגרמה או אנלוגיה ויזואלית (למשל "כדור
   מתגלגל במורד גבעה" להסבר Momentum) עוזרות ורצויות.
4. **שורת "מה עושים בפועל" / Best Practice מפורשת** - כלל אצבע, ערך
   טיפוסי, או תשובה ישירה לשאלה שנפתחה בה השקף. זו לא תוספת אופציונלית
   - זה מה שהופך שקף מ"כותרת עם בולטים" לתוכן שלם ועצמאי. דוגמאות
   מהמצגת-לדוגמה: "אם המודל לא עובד - כדאי להקטין את ה-Learning Rate"
   (שקף Learning Rate Decay), "dropout prob = 0.2 ← 20% סיכוי לכל
   צומת" (שקף Dropout).

**קשתות-מושג (Concept Arcs) למושגים מורכבים:** לא כל מושג נכנס בכבוד
לשקף בודד. כשמנגנון דורש הסבר רב-שלבי (בעיה → שם הפתרון → איך זה עובד
→ [נוסחה/דיאגרמה אם רלוונטי] → guidance מעשי), פרסו אותו על פני 2-5
שקפים ברצף במקום לדחוס הכל לשקף אחד או לדלל אותו לשקף שטחי. זה מעדכן
את "כלל השקף הנפרד למושג" מסטנדרט 1: הכלל המחייב האמיתי הוא "אף מושג
לא נעלם או משולב-בלי-הסבר", לא "בדיוק שקף אחד למושג" - מושג מותר ואף
רצוי שיקבל כמה שקפים ברצף כשהוא דורש זאת (למשל: "Workflow" או
"Hallucination" בקורס הזה יכולים להיפרס לקשת של 2-3 שקפים - בעיה,
מנגנון, ומה עושים בפועל - במקום שקף סיכום בודד).

**איך זה משפיע על ספירת השקפים:** קשתות-מושג באופן טבעי ידחפו חלק
מהשיעורים מעל 16-20 שקפים (ראו סטנדרט 3) - זה תקין ורצוי, ולא צריך
"לקצר בכוח" כדי לחזור למספר קבוע. המדד האמיתי הוא: כל שקף תוכן עומד
בפני עצמו לפי שלושת הרכיבים למעלה, לא ספירת שקפים שרירותית.

**כלל בדיקה מחייב: אסור להשאיר שאלה שנפתחה בשקף בלי תשובה מפורשת על
אותו שקף, ואסור לכלול רשימת שלבים/מונחים בלי הסבר לכל פריט.** זו לא
מצגת משקיעים (שבה מותר לרמוז ולהשאיר את הקהל עם שאלות פתוחות) - זו
מצגת קורס, וסטודנט שקורא אותה חייב לצאת עם תשובה מלאה, לא עם עוד שאלה.
שתי דוגמאות קונקרטיות מתיקון בפועל בשיעור 1, שממחישות את הכלל:
- שקף עם "השאלה: למה בכלל ללמוד היסטוריה של כלים?" ואז בולטים שרק
  מתארים את השרשרת ההיסטורית בלי לחבר בחזרה לשאלה - **פסול.** התיקון:
  בולט ראשון מפורש שמתחיל ב"**התשובה:**" ועונה ישירות על השאלה, לפני
  שעוברים לפרטים התומכים.
- שקף "Workflow" שהציג רשימה ממוספרת של 10 שלבים (1. הבנת הבעיה
  2. Specification 3. בניית Context...) בלי מילה על מה כל שלב אומר
  בפועל - **פסול**, גם אם יש שקף פתיחה+שאלה+best practice סביב הרשימה.
  התיקון: כל פריט ברשימה מקבל משפט הסבר צמוד ("**Specification** —
  כותבים מסמך קצר שמתאר מה בונים ולמה"), ומכיוון שרשימה של 10 שלבים
  מוסברים לא נכנסת בפונט קריא לשקף אחד - זו קשת-מושג של 2 שקפים
  (שלבים 1-5, שלבים 6-10), לא שקף אחד דחוס.

**כלל נוסף: נתון השוואתי חייב בסיס השוואה מפורש על השקף עצמו.** "איטיים
ב-19%" או "פי 1.7 יותר בעיות" הם חסרי משמעות בלי לדעת מול מה משווים.
טעות שקרתה בפועל בשיעור 1: שקף METR אמר "בפועל: איטיים ב-19%" בלי
לציין "לעומת מה" (לעומת עבודה על אותן משימות בלי כלי AI בכלל) - ושקף
DORA אמר "PR עם AI = פי 1.7 יותר בעיות" בלי לציין "יותר מ-PR אנושי".
כל נתון מהצורה "X% יותר/פחות/מהיר/איטי" חייב את בסיס ההשוואה באותו
משפט, לא כהנחה מרומזת.

**כלל נוסף: מספר שמחושב מנתון קודם (לא מוצג כפי שהוא במקור) חייב
להראות את החישוב או להפנות במפורש לנתון המקורי.** טעות שקרתה בפועל
בשיעור 1: שקף סיכום כתב "71% לא סומכים על הפלט" כשבשקף מוקדם יותר
נכתב "רק 29% סומכים על הפלט" - 71% הוא בעצם 100 פחות 29, אבל זה
מעולם לא הוצג כחישוב. סטודנט שקורא רק את שקף הסיכום מקבל מספר חדש
משום מקום. התיקון: "כפי שראינו - רק 29% סומכים על הפלט (כלומר 71%
לא)" - מחבר במפורש בין השקפים במקום להציג מספר גזור כעובדה עצמאית.

**מונחים שמוזכרים לפני שקף ההסבר המלא שלהם** (למשל "Pull Request" או
"Deploy" מוזכרים בשיעור 1 אך מוסברים לעומק רק בשיעור 2) - חייבים גלוסה
קצרה בסוגריים/במשפט בפעם הראשונה שהם מופיעים (כמו "בקשות שילוב קוד
(Pull Requests)"). **שימו לב: הפניה עתידית ("נסביר על Git בשיעור הבא")
היא תוספת מותרת, אבל אף פעם לא תחליף לגלוסה המיידית.** טעות שקרתה בפועל
בשיעור 1: שקף כתב "IDE + Autocomplete → Git" ואז רק הערה "את IDE, Git
ו-Autocomplete נסביר לעומק בשיעור הבא" - בלי אף מילת הסבר עכשווית. זה
פסול, כי הסטודנט נשאר עם מונחים ריקים לאורך שקף שלם (ואז גם שקף ציר-זמן
נוסף שחוזר על אותם מונחים בלי שום הסבר). התיקון: גלוסה קצרה-קצרה תמיד
עכשיו (למשל "**IDE** (תוכנה לכתיבה, הרצה ובדיקה של קוד)"), וההפניה
לשיעור הבא היא רק לציון שההעמקה תגיע שם - לא תירוץ לדלג על ההסבר הבסיסי.
הכלל הזה חל על **כל** קיצור/ראשי-תיבות טכני שמופיע כתוכן שקף: IDE, API,
CLI, Context, PR, וכדומה - לא רק על שמות חברות.

**אותו כלל חל על שמות חברות, ארגונים, כלים ומחקרים - לא רק מונחים
טכניים.** טעות שקרתה בפועל בשיעור 1: שקף אמר "מערכת 'Minions' הפנימית
של Stripe" בלי להסביר בכלל מה זה Stripe (חברת תשלומים דיגיטליים
עולמית) או מה זה Minions (המערכת הפנימית שהחברה בנתה) - הנחה שגויה
שהסטודנטים "כבר מכירים". **ההנחה חייבת להיות הפוכה: הסטודנטים לא
מכירים אף שם חברה, ארגון מחקר, כלי, או סקר שמוזכר בשקף - כולל שמות
"מוכרים" לכאורה.** כל שם כזה שמופיע כחלק מתוכן שקף (כותרת, בולט, נתון)
- לא רק כציון מקור קטן בתחתית השקף - צריך גלוסה קצרה בסוגריים בפעם
הראשונה שהוא מופיע. דוגמאות מהתיקון בפועל:
- "Stripe (חברת תשלומים דיגיטליים עולמית)"
- "'Minions' - מערכת AI פנימית שStripe בנתה לפיתוח אוטומטי"
- "METR (ארגון מחקר עצמאי שבוחן ביצועים אמיתיים של כלי AI)"
- "דו\"ח DORA (מחקר שנתי מוביל בתעשייה על תהליכי פיתוח, מבית Google)"
- "TELUS (חברת תקשורת קנדית גדולה)"
- "Zapier (פלטפורמה לחיבור אוטומטי בין אפליקציות)"

חריג: ציוני מקור בשורת "מקור:" הקטנה בתחתית השקף (כמו "מקור: Forbes
2026") הם ביבליוגרפיה, לא תוכן טעון-משמעות - **אלה לא צריכים גלוסה**.
ההבחנה: אם השם מופיע בתוך כותרת/בולט/משפט שהסטודנט אמור להבין ולזכור -
הוא צריך גלוסה. אם הוא מופיע רק כציטוט מקור בשורה קטנה - לא.

**מצגת כמקור יחיד - לא "ראשון מבין כמה", יחיד ממש:** בעקבות טעות
שחזרה על עצמה בבניית שיעור 1, המבנה השתנה: **אין יותר lesson.md,
exercises.md, demo/, או diagrams.md בתיקיית השיעור בכלל** (ראו "מבנה
הפלט לכל שיעור" למעלה). זה לא רק כלל ניסוח - זה שינוי מבנה קבצים
בפועל. לכן השאלה "האם להכניס הסבר/גלוסה/דוגמה/פרומפט לשקף" היא כבר
לא שיקול של "כפילות מול קובץ אחר" - **אין קובץ אחר**. כל תוכן לימודי
(לא מנהלתי) שהסטודנט צריך כדי להבין או לבצע חייב להיות על השקף עצמו
במלואו, כי פשוט אין לאן אחר לשים אותו.

**כלל מחייב: אסור הפניה לקובץ אחר על תוכן שהסטודנט או המרצה צריכים
בזמן אמת.** טעות שקרתה בפועל בשיעור 1, פעמיים: (א) שקף "הדגמה חיה"
הציג רק את הפרומפט הראשון מתוך שלושה, ולפרומפטים 2-3 כתב "ראו
demo/prompt.md" - המרצה נאלץ לעזוב את המצגת באמצע ההדגמה כדי לפתוח
קובץ אחר; (ב) שקף התרגול הפנה ל-exercises.md במקום להכיל את המשימה,
הפרומפט הפתיחתי, וה-Definition of Done. שני התיקונים: כל התוכן הוטמע
במלואו בשקפים (כל אחד התפרס על 2-3 שקפים ברצף לפי עקרון קשת-המושג,
כי לא נכנס בפונט קריא לשקף אחד) - ואז lesson.md/exercises.md/demo/
נמחקו מהתיקייה לגמרי, כי לא נשאר בהם תוכן שלא כבר קיים במצגת. **מבחן
מעשי: אם המרצה או הסטודנט צריכים לפתוח קובץ נוסף כדי להמשיך לעקוב
אחרי השיעור - זה כשל בבניית השקף.** הביטויים "ראו X.md", "בקובץ X.md",
"לפרטים נוספים ר'" **אסורים** בשקפים לתוכן מהותי (פרומפטים, הסברים,
דוגמאות, שלבים, תרגילים). יוצא מן הכלל היחיד: הפניות מנהלתיות טהורות
שאינן תוכן לימודי לסטודנט - הפניה מתוך instructor_notes.md/solutions.md
חזרה לשקף מסוים ("ראו שקף 12") מותרת, כי אלה קבצים למרצה בלבד, לא
לסטודנט; וכך גם "פירוט מלא: references.md" בשקף המקורות בסוף המצגת,
כי זה גיבוי לציטוט שכבר קיים בתמצית על השקף עצמו, לא תוכן חסר.

### 3. סטנדרט צפיפות תוכן ואינטראקטיביות

לקח מבניית שיעורים 1-2: תוכן שנכתב רק כרשימת בולטים תמציתית "נגמר"
בפועל הרבה לפני הזמן שהוקצה לו (סטודנט קורא בולט ב-15 שניות; מרצה
שמסביר בולט בלי תוספת אמיתית מסיים תוך כמה דקות, לא תוך רבע שעה). כדי
שכל שיעור באמת ימלא את 90 הדקות שהוקצו לו:

1. **כל בלוק תוכן צריך פעילות אינטראקטיבית אחת לפחות** (שאלת כיתה +
   זמן לתשובות, פעילות בזוגות של 2-3 דקות, או "הרימו יד"). פעילויות
   תמיד לוקחות יותר זמן אמיתי מהסבר פרונטלי - זה לא "בזבוז זמן", זו
   הדרך האמינה למלא זמן הרצאה נכון.
2. **כל שיעור צריך לפחות "נקודת גישור" אחת למסלול בינה מלאכותית** -
   בולט מפורש על שקף רלוונטי (בסגנון "**למסלול AI:**...", ראו דוגמה
   בשקף Hallucination בשיעור 1) שמקשר את הנושא לרקע שהסטודנטים כבר
   מביאים מהמסלול שלהם (Colab/Kaggle notebooks, שימוש ב-ChatGPT/Claude
   לקוד עזר מחקרי, מושגי ML/AI רלוונטיים) - ומשתקף גם ב-instructor_
   notes.md כהנחיה למרצה להרחיב עליו בעל-פה. זה גם ממלא זמן אמיתי
   (הסטודנטים מזהים את עצמם בדוגמה, יש להם מה להגיד) וגם הופך את
   הקורס לרלוונטי יותר לקהל הספציפי הזה.
3. **בכל נושא שיש לו יותר מדוגמה אחת בעולם האמיתי - תנו שתיים.**
   לדוגמה: בשיעור 1 מקרה הבוחן Gemini CLI→Antigravity קיבל דוגמה שנייה
   (שינוי מסלול GitHub Copilot Student) בשקף נפרד, כי דוגמה אחת בלבד
   לא מילאה את הזמן שהוקצה לה. חפשו באינטרנט דוגמאות עדכניות נוספות
   במקום להסתפק בדוגמה יחידה כשיש זמן שצריך למלא.
4. **בסוף כל instructor_notes.md, כללו "חלוקת זמן מעודכנת"** - פירוט
   מדויק של כמה דקות לכל בלוק תוכן, שמסתכם לסביבות 90 (מותר 90-95, לא
   פחות מ-85). אם הסכום נמוך משמעותית מ-90 - זה סימן שצריך להוסיף עוד
   פעילות/דוגמה/נקודת גישור, לא רק "לדבר לאט יותר".
5. **אל תוסיפו תוכן על ידי הרחבת כל בולט לפסקה ארוכה** - זה עובד הפוך:
   שקפים עמוסים בטקסט גרועים פדגוגית. התוכן הנוסף שייך לפעילויות,
   לדוגמאות נוספות, ולשקפים/קשתות-מושג נוספים במצגת עצמה (אין עוד
   lesson.md לפרוק אליו) - לא לעומס טקסט על השקף עצמו. יחד עם זאת, כל
   שקף חייב לעמוד בסטנדרט 2 (שקף עצמאי) - "קצר" לא אומר "חסר תוכן".
6. **כל שיעור צריך 16-20 שקפים לפחות** (לא 10-12), ולפעמים יותר
   כשקשתות-מושג (סטנדרט 2) דורשות זאת - כדי למלא 90 דקות הרצאה בעומק
   אמיתי. הדרך הנכונה להגיע לזה היא לא לנפח בולטים קיימים, אלא להוסיף
   שקפי תוכן חדשים מהסוגים הבאים:
   - **שקפי סטטיסטיקה** (Stat Callouts - מספר גדול + תווית קטנה) עם
     נתונים אמיתיים ומצוטטים - חפשו באינטרנט נתוני שוק, סקרים,
     ובנצ'מארקים עדכניים (DORA, Stack Overflow Developer Survey, METR,
     SWE-bench/Terminal-Bench, נתוני אימוץ של כלים) לפני שכותבים שקף
     כזה. כל שקף נתונים חייב ציון מקור בתחתית השקף.
   - **שקפי מקרה בוחן נוספים** - כשיש יותר מדוגמה אחת בעולם האמיתי
     לנושא, תנו שתיים (ראו שיעור 1: מקרה Gemini CLI + מקרה Copilot
     Student; שיעור 1: Stripe/TELUS/Zapier).
   - **שקפי "הנתונים תומכים בזה"** שמקשרים בין קונספט שכבר לומד
     (Workflow, Best Practices) לבין נתון אמיתי שמוכיח אותו במספרים -
     אלה השקפים הכי אפקטיביים פדגוגית כי הם הופכים טענה לעובדה מגובה.
   - **שקף מקורות** בסוף כל שיעור (לפני/אחרי הסיכום) שמרכז את כל
     המקורות שצוטטו.
   - חשוב: אין לבדות נתונים. כל מספר על שקף חייב להגיע מחיפוש אמיתי
     באינטרנט ולהיות מתועד ב-references.md. אם לא מוצאים מקור אמין
     לנתון - לא כותבים אותו כעובדה, לכל היותר כהערכה מנוסחת בזהירות.
   - כשמציגים בנצ'מארקים טכניים (כמו SWE-bench) - תמיד לכלול אזהרת
     מתודולוגיה קצרה (שיטות מדידה שונות, שונות בין הרצות) כדי לא
     להציג מספר בודד כאמת מוחלטת. זה גם תוכן וגם מודלים חשיבה ביקורתית.

## דרישות מדויקות לכל קובץ בתיקיית השיעור

**slides.md (+ build_deck.js / lesson_XX_slides.pptx)** - תוכן המצגת
בפועל, **המקור היחיד לסטודנט**. **16-20 שקפים לפחות, ויותר כשקשתות-
מושג דורשות זאת, ועוד יותר כשמכניסים לתוכה גם את התרגול וההדגמה
במלואם** (ראו "סטנדרט צפיפות תוכן" ו"סטנדרט שקף עצמאי" למעלה - שניהם
לא אופציונליים): שקף פתיחה (מספר שיעור, כותרת, 3-4 שאלות מנחות) →
שקפי תוכן, **כל אחד עומד בפני עצמו** לפי סטנדרט שקף עצמאי (מושג/בעיה
→ הסבר אמיתי → best practice מפורש), ומושגים מורכבים מפורסים על פני
כמה שקפים ברצף (קשת-מושג) במקום שקף בודד ושטחי → שקפי נתונים/סטטיסטיקה
מבוססי מקורות אמיתיים משולבים בין שקפי התוכן → **שקף/י הדגמה חיה עם
הפרומפט/ים המדויקים מוטמעים בתיבת טקסט על השקף עצמו** (לא "ראו קובץ
נפרד") → **שקף/י תרגול עם מטרה, דרישות המשימה, פרומפט פתיחה, ו-
Definition of Done מוטמעים במלואם** → שקף סיכום (3 Takeaways + חיבור
לשבוע הבא, כולל טעימה קצרה מהמשך הקורס) → שקף מקורות. כל דיאגרמה
נדרשת מתוארת כ-`[תמונה: ...]` placeholder בתוך השקף הרלוונטי עצמו -
לא בקובץ נפרד; אם דיאגרמה מורכבת (flowchart, ציר זמן), אפשר לתאר את
המבנה שלה במילים בתוך אותו placeholder כדי שמי שמכין את האיור בפועל
ידע מה לצייר. סמן כל שקף עם `---` כמפריד (תואם Marp).

**instructor_notes.md** - מדריך הפעלה למרצה בלבד, מקביל לכל שקף ב-
slides.md: מה להגיד/להרחיב, תזמון משוער לכל בלוק, שאלות לדיון שכדאי
לזרוק לכיתה, ומוקשים נפוצים (איפה סטודנטים בד"כ נתקעים או מבינים לא
נכון). זה קובץ שהסטודנט לא רואה, אז אין בעיה שהוא מפנה חזרה לשקפים
("ראו שקף X") - ההבדל בין זה ל-slides.md הוא בדיוק שזה תלוי בקהל
(מרצה) ולא בקהל (סטודנט).

**solutions.md** - לא פתרון יחיד "נכון", למרצה בלבד (לא לחלוקה מראש

#    קודם (כמו 100 פחות 29) מציג את החישוב ולא מוצג כעובדה עצמאית חדשה
grep -n '%' slides.md

## הערה טכנית קריטית: RTL אמיתי ב-build_deck.js (pptxgenjs)

**באג ידוע ומתועד ב-pptxgenjs: כשמעבירים ל-`addText` מערך של כמה
"ריצות טקסט" (rich text runs - כל בולט/שאלה/best-practice שבנוי מכמה
`{ text, options }`), האופציה `rtlMode: true` שמוגדרת ב-options הכלליים
של קריאת `addText` לא משפיעה בכלל.** pptxgenjs קורא rtlMode רק מתוך
`options` של הריצה **הראשונה** במערך (`line[0].options.rtlMode`). התוצאה:
המצגת נראית תקינה ב-QA (רינדור דרך LibreOffice/soffice.py, שמזהה RTL
אוטומטית לפי התווים) אבל נפתחת Left-to-Right בפועל ב-PowerPoint האמיתי,
כי חסר `rtl="1"` ב-XML. זה קרה בפועל בשיעורים 1 ו-2 ותוקן.

**חובה:** בכל פונקציית עזר שבונה מערך ריצות טקסט (בולטים, שאלה, best
practice, כל טקסט עם יותר מריצה אחת) - `rtlMode: true` חייב להיות בתוך
ה-`options` של **כל ריצה וריצה** במערך, לא רק בתוך אובייקט ה-options
הכללי שמועבר ל-`addText`. הדרך הבטוחה ביותר: לבנות helper מרכזי (כמו
`runs()`/`bulletRuns()` בשיעור 1) שמזריק `rtlMode: true` אוטומטית לתוך
כל ריצה שהוא מייצר, ולהשתמש רק בו - לא לבנות מערכי ריצות טקסט "ידנית"
בכל שקף בנפרד.

**בדיקת אימות מחייבת לפני שמסיימים שיעור:** אחרי `node build_deck.js`,
לפרוס את קובץ ה-pptx (`unzip`) ולוודא שכל (או כמעט כל, פרט למספרים/
אחוזים שהם ניטרליים מבחינת כיוון) בלוקי `<a:pPr>` ב-XML מכילים
`rtl="1"`. פקודה לדוגמה:
```
unzip -o -q lesson_XX_slides.pptx -d /tmp/check
grep -o "<a:pPr[^>]*>" /tmp/check/ppt/slides/slide*.xml | wc -l
grep -o "<a:pPr[^>]*rtl=\"1\"[^>]*>" /tmp/check/ppt/slides/slide*.xml | wc -l
```
שני המספרים צריכים להיות קרובים מאוד (פער מותר רק עבור טקסט שהוא מספר/
אחוז טהור). רינדור חזותי דרך LibreOffice **לא מספיק** לבדיקה הזו - הוא
לא חושף את הבאג כי הוא מזהה RTL אוטומטית גם בלי `rtl="1"` ב-XML.

## הערה טכנית קריטית שנייה: מיקום בולט/סוגריים ב-PowerPoint אמיתי כשבולט מתחיל באנגלית

**באג נוסף שהתגלה בפועל בשיעור 1, אחרי שבאג ה-rtl="1" כבר תוקן:**
כשבולט מתחיל במונח מודגש באנגלית (למשל "**LLM (Large Language Model)**
= תוכנה שאומנה...") והבולט "•" הוטמע כתו טקסט מילולי שהודבק לתחילת
הריצה הראשונה (`r[0].text = "• " + r[0].text`) - ב-PowerPoint אמיתי
(לא ב-LibreOffice) מנוע ה-bidi עלול לסדר לא נכון תווים ניטרליים שנמצאים
על הגבול בין ריצת טקסט אנגלית לעברית: הבולט "•", הסוגריים, וסימן "="
"מתניידים" למקום הלא נכון חזותית - גם כש-`rtl="1"` מוגדר נכון ברמת
הפסקה. שוב, LibreOffice לא חושף את זה כי הוא לא משחזר במדויק את מנוע
ה-bidi של PowerPoint.

**התיקון:** לא להטמיע "•" כתו טקסט בתוך ריצה. להשתמש בתכונת ה-bullet
המובנית של PPTX (`buChar` ברמת פסקה) - זה עיצוב פסקה, לא תוכן טקסט,
ולכן לא כפוף לחלוטין לרזולוציית ה-bidi של תוכן הטקסט:
```js
r[0] = {
  ...r[0],
  options: { ...r[0].options, bullet: { code: "2022", indent: 18 } },
};
// בלי "• " + r[0].text
```

**בדיקת אימות:** אחרי `unzip`, לוודא ש-(א) כל פסקת-בולט מכילה
`<a:buChar char="&#x2022;"/>` בתוך ה-`<a:pPr>` שלה, ו-(ב) אין אף תו "•"
מילולי שנשאר בתוך `<a:t>` (`grep -c '•' slideN.xml` צריך להיות 0).

**אזהרה חשובה: LibreOffice לא תמיד מרנדר `buChar` ב-RTL בעקביות** -
בבדיקה נצפה ש-LibreOffice הציג בולטים על חלק מהפסקאות ולא על אחרות,
למרות שה-XML זהה לחלוטין בין כולן (מאומת ידנית). זהו כשל ידוע של
LibreOffice ברינדור בולטים ב-RTL, לא כשל במבנה ה-XML. **אי אפשר לסמוך
על תצוגת ה-QA של LibreOffice כדי לאשר שהבולטים מוצגים נכון** - יש
לבקש מהמשתמש אימות מפורש ב-PowerPoint אמיתי אחרי כל שינוי שנוגע לבולטים,
בדיוק כמו עם באג ה-rtl="1" המקורי.
